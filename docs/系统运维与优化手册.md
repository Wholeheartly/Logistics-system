## 1. 环境差异排查

### 1.1 环境对比总览

| 配置项 | 本地开发环境 | 生产环境 (ECS) | 差异说明 |
|--------|-------------|---------------|----------|
| **操作系统** | Windows 11 | Ubuntu (Linux) | ⚠️ 跨平台需注意路径分隔符 |
| **数据库** | SQLite (`logistics.db`) | PostgreSQL 16 Alpine | ⚠️ **关键差异**，SQL语法/类型有区别 |
| **DATABASE_URL** | `sqlite:///./logistics.db` | `postgresql://logistics:***@db:5432/logistics` | 生产使用环境变量 |
| **Python** | 本地安装 | 3.12 (Docker容器) | 版本一致 |
| **FastAPI** | 0.115.12 | 0.115.12 | ✅ 一致 |
| **SQLAlchemy** | 2.0.40 | 2.0.40 | ✅ 一致 |
| **Node.js** | 本地安装 | Docker容器内构建 | 构建环境隔离 |
| **前端部署** | `localhost:5173` (Vite dev) | EdgeOne Pages CDN + ECS Nginx | ⚠️ 双入口架构 |
| **API地址** | `http://localhost:8000` | `http://47.114.80.214` | 生产暴露IP |
| **HTTPS** | 无 | EdgeOne Pages有, ECS无 | ⚠️ ECS未配置SSL |
| **CORS** | `localhost` | 多域名含ECS IP | 生产白名单更宽 |
| **容器化** | 无 | Docker Compose (3容器) | 生产完全容器化 |
| **日志** | 控制台 | `/opt/logistics-system/deploy/logs/` | 生产持久化 |
| **数据卷** | 本地文件 | Docker Volume (`pgdata`, `uploads`) | 生产数据持久化 |

### 1.2 关键差异分析

#### 差异1：数据库引擎不一致 (已导致问题)

- **问题**：`init_users.py` 和 `seed_database.py` 硬编码 SQLite 路径，生产环境使用 PostgreSQL
- **影响**：部署后用户表为空，登录失败
- **已修复**：`init_users.py` 已改为读取 `DATABASE_URL` 环境变量
- **待修复**：`seed_database.py` 仍硬编码 SQLite，需同样改造

#### 差异2：前端双入口架构

```
用户 → EdgeOne Pages (CDN) → 前端静态资源
                              ↓ API请求
         ECS (47.114.80.214) → Nginx → FastAPI
```

- EdgeOne Pages 部署的前端通过 `VITE_API_BASE=http://47.114.80.214` 直接请求ECS
- ECS 的 Nginx 同时提供前端静态文件和API代理
- **风险**：ECS IP暴露，无HTTPS加密

#### 差异3：环境变量管理

- 本地：`.env.example` 模板，默认 SQLite
- 生产：`deploy/.env.production`，PostgreSQL + 密钥
- **注意**：`seed_database.py` 不读取环境变量，无法在生产环境执行数据导入

### 1.3 环境一致性改进建议

1. **统一数据库**：本地开发也应使用 PostgreSQL (Docker)，避免 SQLite/PostgreSQL 差异
2. **环境变量模板**：`.env.example` 应包含 PostgreSQL 配置示例
3. **脚本兼容性**：所有数据库脚本必须支持 `DATABASE_URL` 环境变量

### 4.2 系统监控指标与异常处理

#### 监控指标

| 指标 | 正常范围 | 警告阈值 | 危险阈值 | 检查方式 |
|------|---------|---------|---------|---------|
| API响应时间 | < 500ms | > 3s | > 5s | `curl -o /dev/null -w "%{time_total}" http://localhost/api/health` |
| HTTP状态码 | 200 | 4xx | 5xx | `curl -o /dev/null -w "%{http_code}" http://localhost/api/health` |
| 磁盘使用率 | < 60% | > 80% | > 90% | `df -h /` |
| 可用内存 | > 2GB | < 500MB | < 200MB | `free -h` |
| 容器状态 | running | restarting | exited | `docker compose ps` |
| 数据库连接数 | < 10 | > 20 | > 30 | `docker compose exec db psql -c "SELECT count(*) FROM pg_stat_activity;"` |
| Docker磁盘占用 | < 3GB | > 5GB | > 8GB | `docker system df` |

#### 异常处理流程

```
异常发现
  ├── 自动告警 (Prometheus alerts.yml)
  │     └── 邮件通知 → 运维人员
  │
  ├── 手动检查
  │     ├── 1. 确认异常现象
  │     ├── 2. 查看日志: tail -f /opt/logistics-system/deploy/logs/nginx/error.log
  │     ├── 3. 检查容器: docker compose ps && docker compose logs --tail=50
  │     └── 4. 检查资源: free -h && df -h && docker stats
  │
  └── 处理措施
        ├── 服务不可用 → docker compose restart <service>
        ├── 磁盘满 → 清理日志/Docker镜像
        ├── 内存不足 → 重启容器/扩容
        └── 数据库异常 → 从备份恢复
```

## 5. 系统更新流程

### 5.1 代码变更管理规范

#### 分支策略

```
main (生产分支)
  ├── develop (开发分支)
  │     ├── feature/xxx (功能分支)
  │     ├── fix/xxx (修复分支)
  │     └── refactor/xxx (重构分支)
  └── hotfix/xxx (紧急修复分支)
```

- `main`：生产环境代码，仅通过 PR 合并
- `develop`：开发集成分支
- `feature/*`：新功能开发
- `fix/*`：Bug修复
- `hotfix/*`：生产紧急修复，从 `main` 拉出，合并回 `main` 和 `develop`

#### 代码审查流程

1. 开发者在功能分支完成开发
2. 创建 Pull Request 到 `develop`
3. 至少1人审查代码
4. CI 通过（lint + test + build）
5. 审查通过后合并

### 5.2 测试环境验证步骤

```bash
# 1. 本地测试
cd backend && pytest -v
cd frontend && npm run lint && npm run test && npm run build

# 2. Docker构建测试
docker compose build

# 3. 功能验证
curl http://localhost/api/health
# 在浏览器中测试前端功能

# 4. 数据库迁移测试（如有）
docker compose exec backend python scripts/migrate_product_fields.py
```

### 5.3 生产环境部署流程

#### 标准部署流程

```bash
# === 在本地开发机执行 ===

# 1. 同步代码到服务器
rsync -avz --exclude 'node_modules' --exclude '__pycache__' --exclude '.git' \
  ./ root@47.114.80.214:/opt/logistics-system/

# === 在ECS服务器执行 ===
ssh root@47.114.80.214

# 2. 进入项目目录
cd /opt/logistics-system

# 3. 备份数据库
docker compose exec -T db pg_dump -U logistics logistics | gzip > deploy/backup/db_pre_deploy_$(date +%Y%m%d_%H%M%S).sql.gz

# 4. 停止服务
docker compose down

# 5. 重新构建并启动
docker compose --env-file deploy/.env.production up -d --build

# 6. 等待服务就绪
sleep 30

# 7. 验证部署
docker compose ps
curl http://localhost/api/health
curl http://localhost/

# 8. 检查日志
docker compose logs --tail=20 backend
docker compose logs --tail=20 frontend
```

#### 回滚方案

```bash
# 方案A：回退代码版本
cd /opt/logistics-system
git log --oneline -5  # 找到上一个稳定版本
git checkout <commit-hash>
docker compose --env-file deploy/.env.production up -d --build

# 方案B：恢复数据库备份
cd /opt/logistics-system
docker compose exec -T db psql -U logistics -d logistics -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
gunzip -c deploy/backup/db_pre_deploy_YYYYMMDD_HHMMSS.sql.gz | docker compose exec -T db psql -U logistics -d logistics

# 方案C：使用之前的Docker镜像
docker images  # 找到之前的镜像tag
# 修改 docker-compose.yml 使用特定镜像tag
docker compose up -d
```
